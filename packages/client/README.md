# graph-italia-cli

A generated TypeScript/axios API client for the Graph Italia server (`packages/server`). It gives you a typed function for every REST endpoint exposed by the API (auth, charts, dashboards, datasources, KPI groups, orgs, projects, API keys, admin), plus the request/response TypeScript types generated from the OpenAPI spec.

The client itself contains **no hand-written request code** — everything under `src/client.ts` and `src/model/` is generated by [Orval](https://orval.dev/) from `graphitalia-openapi.yml`. Do not edit those files by hand; regenerate them instead (see below).

## Package layout

```
packages/client/
├── graphitalia-openapi.json   # OpenAPI spec, exported from the server (source of truth)
├── graphitalia-openapi.yml    # YAML conversion of the same spec, used as Orval's input
├── orval.config.ts            # Orval codegen configuration
├── rollup.config.js           # Bundles src/ -> dist/ (CJS + ESM + .d.ts)
├── src/
│   ├── index.ts               # Public entry point, re-exports the generated client + models
│   ├── client.ts               # GENERATED — one function per API operation
│   └── model/                  # GENERATED — one file per request/response type
├── usage.ts                    # Runnable example against dist/index.js
├── .env.sample / .env          # API_BASE_URL / API_KEY for usage.ts
└── dist/                       # Build output (gitignored)
```

## Installing as an npm module

This package is **not currently published to the npm registry** — the repo's `release.yml` CI workflow only auto-publishes `packages/components` (`graph-italia-components`) on tagged releases. `packages/client` has a `build` script and is shaped to be published the same way, but nobody has wired that job up yet.

Until that changes, use one of these instead:

1. **Inside this monorepo** (recommended): any other workspace package can depend on it directly via Bun/npm workspaces:
   ```jsonc
   // some other package.json
   "dependencies": {
     "graph-italia-cli": "workspace:*"
   }
   ```
   Then `bun install` from the repo root will symlink it. Consumers importing from `dist` (the package's `main`/`module` entry points) need it built at least once — either `bun run build` from the root (builds every workspace package) or `cd packages/client && bun run build`.

2. **As a standalone tarball**, e.g. for an external project:
   ```bash
   cd packages/client
   bun run build
   npm pack        # produces graph-italia-cli-<version>.tgz
   # in the consuming project:
   npm install /path/to/graph-italia-cli-<version>.tgz
   ```

3. **Publish it yourself** to a registry you control (npmjs, GitHub Packages, a private registry):
   ```bash
   cd packages/client
   bun run build
   npm publish --access public   # or --registry <your-registry>
   ```
   If you do this regularly, consider copying the `publish-npm` job from `.github/workflows/release.yml` (currently scoped to `packages/components`) and pointing a copy at `packages/client`.

Once installed, consume it like any other npm package:

```ts
import axios from "axios";
import { getGraphItaliaAPI } from "graph-italia-cli";

const axiosInstance = axios.create({
  baseURL: "https://your-graph-italia-server.example.com",
  headers: { Authorization: `Bearer ${API_KEY}` },
});

const client = getGraphItaliaAPI(axiosInstance);
const charts = await client.getApiCharts();
```

## Building

```bash
cd packages/client
bun install
bun run build
```

- `rollup.config.js` compiles `src/index.ts` → `dist/index.js` (CJS), `dist/index.esm.js` (ESM), and `dist/types/*.d.ts` (declarations), per the `main`/`module`/`types` fields in `package.json`.
- `axios` is an external peer — it is **not** bundled, so consumers must have `axios` installed themselves.

## Regenerating the client after the API changes

The client is derived from the server's OpenAPI spec, so the workflow is: **update the server → export its spec → regenerate this package → rebuild.**

1. Make sure `packages/server` is running (or that its OpenAPI spec generation is up to date — it's served at `/openapi.json` via `hono-openapi`/Scalar, see `CLAUDE.md`).
2. Fetch the latest spec and drop it into this package as `graphitalia-openapi.json` (e.g. `curl http://localhost:3003/openapi.json -o graphitalia-openapi.json`, adjusting the base URL/prefix to match your server's `ROUTES_PREFIX`).
3. Convert it to YAML (Orval is configured to read the `.yml` file):
   ```bash
   bun run convert   # json2yaml graphitalia-openapi.json > graphitalia-openapi.yml
   ```
4. Regenerate the typed client and models:
   ```bash
   bun run generate  # orval --config orval.config.ts
   ```
   This rewrites `src/client.ts` and everything under `src/model/`, then runs Prettier on the changed files (`hooks.afterAllFilesWrite` in `orval.config.ts`).
5. Rebuild the distributable bundle:
   ```bash
   bun run build
   ```
6. Review the diff. New/changed endpoints show up as new/changed exported functions in `src/client.ts` and new/changed types in `src/model/`; review call sites elsewhere in the monorepo (webapp, other clients) that consume this package.

**Never hand-edit `src/client.ts` or files in `src/model/`** — they're overwritten wholesale on the next `bun run generate`. If the generated output is wrong, fix it at the source: the server route's `describeRoute`/Zod schema, then regenerate.

### Maintaining/updating checklist

- Adding a new server route or changing a request/response shape → regenerate (steps above); no manual client changes needed.
- Renaming an operation ID on the server changes the generated function name in `src/client.ts` — this is a breaking change for consumers, bump the package's `version` in `package.json` accordingly.
- Upgrading Orval itself (`devDependencies.orval` in `package.json`) can change output conventions (naming, file layout) — regenerate and diff carefully after a version bump.

## Functions provided (`getGraphItaliaAPI(axiosInstance)`)

Calling `getGraphItaliaAPI(axiosInstance)` returns an object with one method per API operation. `axiosInstance` is optional and defaults to the global `axios` — pass your own instance (as in the examples above) to set `baseURL`, default headers (auth), interceptors, etc.

| Area | Functions |
|---|---|
| **Admin** (session, ADMIN role) | `getApiAdminUsers`, `deleteApiAdminUsersById`, `postApiAdminUsersByIdActivate`, `postApiAdminUsersByIdResendActivation`, `postApiAdminUsersByIdResetPassword` |
| **Auth** | `getApiAuthUser`, `postApiAuthRegister`, `postApiAuthLogin`, `postApiAuthRecover`, `postApiAuthResend`, `getApiAuthLogout`, `postApiAuthVerify`, `getApiAuthConfirmByUidByCode`, `postApiAuthResetPassword`, `putApiAuthPwd` |
| **API keys** (session only) | `getApiApikeys`, `postApiApikeys`, `deleteApiApikeysById`, `patchApiApikeysByIdRevoke`, `patchApiApikeysByIdReinstate`, `getApiApikeysByIdLogs` |
| **Charts** | `getApiCharts`, `postApiCharts`, `getApiChartsById`, `deleteApiChartsById`, `putApiChartsById`, `getApiChartsShowById` (public), `postApiChartsPublishById`, `postApiChartsKpiGroup` (session only), `putApiChartsKpiGroupById`, `getApiChartsKpiGroupById` |
| **Dashboards** | `getApiDashboards`, `postApiDashboards`, `getApiDashboardsById`, `deleteApiDashboardsById`, `putApiDashboardsById`, `getApiDashboardsShowById` (public), `putApiDashboardsByIdSlots` |
| **Data sources** | `getApiDatasources`, `postApiDatasources`, `getApiDatasourcesById`, `putApiDatasourcesById`, `deleteApiDatasourcesById`, `getApiDatasourcesByIdLinks`, `postApiDatasourcesByIdLinks`, `deleteApiDatasourcesByIdLinksByChartId` |
| **AI hints** (session only) | `postApiHints` |
| **Orgs** (session only) | `getApiOrgs`, `postApiOrgs`, `getApiOrgsByOrgId`, `putApiOrgsByOrgId`, `deleteApiOrgsByOrgId`, `getApiOrgsByOrgIdMembers`, `postApiOrgsByOrgIdMembers`, `getApiOrgsByOrgIdProjects`, `putApiOrgsByOrgIdMembersByUserId`, `deleteApiOrgsByOrgIdMembersByUserId` |
| **Projects** (session only) | `getApiProjects`, `postApiProjects`, `getApiProjectsPersonal`, `getApiProjectsByProjectId`, `putApiProjectsByProjectId`, `deleteApiProjectsByProjectId`, `getApiProjectsByProjectIdMembers`, `postApiProjectsByProjectIdMembers`, `putApiProjectsByProjectIdMembersByUserId`, `deleteApiProjectsByProjectIdMembersByUserId`, `postApiProjectsByProjectIdOrgs`, `deleteApiProjectsByProjectIdOrgsByOrgId` |

Every function returns `Promise<AxiosResponse<T>>`; a matching `<FunctionName>Result` type alias (e.g. `GetApiChartsResult`) is exported from `src/client.ts` for typing call sites without importing axios types directly. Request/response body shapes live in `src/model/` and are re-exported from the package root.

## Authentication methods

The server (`packages/server/lib/middlewares.ts`) accepts exactly two kinds of credential, read from either the `access_token` cookie or an `Authorization: Bearer <token>` header — the client always uses the header, since it's a Node/browser-agnostic library with no cookie jar of its own:

1. **Session JWT** — obtained from `postApiAuthLogin`. Short-lived (1-day expiry per `CLAUDE.md`), tied to a specific user, and required (not just accepted) for operations that don't make sense for a machine credential: managing orgs/projects/members, API keys themselves, admin actions, AI hints, and creating KPI-group charts.
2. **API key** (`dv_`-prefixed) — obtained from `postApiApikeys` (requires a session JWT to create). Long-lived, scoped to a single project at creation time, and revocable. Accepted anywhere `requireAuth` is used: charts, dashboards, datasources, KPI-group read/update.

There is no separate client-side "auth mode" flag — you simply set whichever credential you have as the Bearer token on your `AxiosInstance`, e.g.:

```ts
const axiosInstance = axios.create({
  baseURL: API_BASE_URL,
  headers: { Authorization: `Bearer ${TOKEN}` }, // session JWT or dv_ API key
});
```

A few endpoints require no auth at all (public, CORS-open on the server): `getApiChartsShowById` and `getApiDashboardsShowById` — these serve published charts/dashboards for embedding.

### API key roles (scopes)

API keys have exactly one of two roles, set at creation (`postApiApikeys`) and enforced server-side:

| Role | Access |
|---|---|
| `READONLY` | Can call GET-style/read endpoints for its scoped project only (`canRead` in `middlewares.ts`). |
| `READWRITE` | Everything `READONLY` can do, plus create/update/delete operations for its scoped project (`canModify` in `middlewares.ts`). |

Roles do **not** stack across projects — a key is bound to exactly one `projectId` (either the caller's default project or an explicit `projectId` passed to `postApiApikeys`), and every request made with that key is scoped to that project regardless of which project the JWT-authenticated equivalent user could otherwise access.

Keys also carry an `expire` (days from creation, default 60) and can be temporarily disabled via `patchApiApikeysByIdRevoke` / re-enabled via `patchApiApikeysByIdReinstate`, without needing to issue a new key.

## Environment variables

Only relevant to `usage.ts` (the example script), not to the library itself — consumers configure the `AxiosInstance` however suits their app.

| Variable | Description | Example (`.env.sample`) |
|---|---|---|
| `API_BASE_URL` | Base URL of the Graph Italia server to call. | `http://localhost:3003` |
| `API_KEY` | Bearer credential — either a `dv_`-prefixed API key or a session JWT. | `dv_6971cb9a_...` |

Copy `.env.sample` to `.env` and fill in real values, then run the example with:

```bash
bun run usage.ts
```

(`.env` is gitignored — never commit real keys.)
